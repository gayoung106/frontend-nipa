초보자 또는 테스트 주도 개발을 모르는 개발자가 개발하는 방식 (보통)
1. 어떤 기능을 구현할지 생각한다(머리로 구상해봄)
2. 실제 기능 구현을 시작
3. 되나 안되나 테스트 해봄
4. 되면 좋고
5. 안되면 우울함 될때까지 한다

- 비효율적일 수 있음

4. 충족하지 못하는 조건이 없을 때까지 한다

React Testing Library에서는 컴포넌트의 return 문 이후 화면에 실제로 렌더링되고 표시되는 내용을 테스트하는 데 중점을 둡니다. 이 접근 방식은 컴포넌트의 내부 구현 세부 사항에만 초점을 두는 대신 사용자의 관점에서 테스트하는 철학과 일치합니다. React Testing Library는 사용자가 응용 프로그램과 상호 작용하는 방식을 가능한 한 정확하게 모방하여 컴포넌트를 테스트할 것을 권장합니다.

React Testing Library를 사용하여 테스트를 작성할 때 일반적으로 다음과 같은 절차를 따릅니다:

React 컴포넌트를 렌더링합니다.
텍스트, 레이블, 속성 또는 기타 사용자가 볼 수 있는 특성을 기반으로 렌더링된 요소를 쿼리합니다.
사용자와 유사한 동작(예: 버튼 클릭, 텍스트 입력)을 사용하여 해당 요소와 상호 작용합니다.
해당 상호 작용의 결과물이나 변경 사항이 발생했는지 확인하는 단언문(assertion)을 추가합니다. 예를 들어 특정 요소가 보이는지 또는 요소의 내용이 변경되는지 여부를 확인할 수 있습니다.
이 접근 방식은 테스트가 컴포넌트의 실제 동작 및 사용자 경험에 집중하도록 하여 구현 세부 사항에 빠지지 않게 합니다. 사용자가 볼 수 있는 동작이 일관되는 한 컴포넌트 구조의 변경에 강한 테스트를 만들어 줍니다.

React Testing Library는 응용 프로그램의 사용자에게 노출되는 측면을 테스트하는 데 특히 적합하지만, 응용 프로그램의 복잡한 또는 중요한 부분의 경우 컴포넌트 내부 로직에 더 깊이 파고들어야 할 수도 있습니다. 그런 경우 React Testing Library와 다른 테스팅 도구나 패턴(예: 유닛 테스트 또는 통합 테스트)을 조합하여 코드베이스의 포괄적인 커버리지를 확보하는 것이 좋습니다.

React bootstrap(프론트엔드 툴킷)
프론트엔드가 하는일: UIUX개발 사용자 인터페이스 및 사용자 경험을 제공하는 일
UIUX를 제공함에 있어서 기본적인 요구는 대체로 비슷함
그래서 기본적인 요구를 만들어서 제공하는 툴킷: 부트스트랩

사용방법: 태그에다가 정해진 클래스명을 쓰기만 하면 부트스트랩 CSS, JS가 적용됨
=> 내가 직접 짠 것보다 훨씬 편하고 빠르게 웹사이트 구성 가능
